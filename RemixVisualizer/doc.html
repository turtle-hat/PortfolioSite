This project was a lot of fun (if I severely underestimated how long it would take) and I think I got something pretty impressive out of it.

As Nick might remember, I was originally going to use the Giphy API instead, since the Scratch API doesn't actually support CORS! I drew a little mockup for how I wanted to improve that here:

Thankfully, I was able to follow along with the PHP resources posted in myCourses to get my AJAX needs fully functioning!

The first thing I worked on was just accessing the JSON returned by my various PHP requests. After that, I came up with a system to recursively request remixes of a project, and after that I just added the relevant returned data for each project into an object, which I could use to build the tree itself.

As you might be able to see from the concept art, I originally planned on having all the projects inside one big grid, but I found it far easier to add each project as its own nested flexbox.

From there, I decided to mainly focus on getting all the settings and interface working and looking nice.

After that, I tidied up the tree by adding borders around remix groups. I'd played around with the idea a little before this, since I didn't want to bit the bullet and figure out how to get individually-drawn lines working, and eventually I realized that by removing the end edge of the border box and thickening the start edge it formed a sort of bracket shape which looks pleasing and expresses relationships well over long distances without looking too cluttered most of the time.

I'm particularly proud of the expanded project information boxes that appear when hovering over a project. It took a while to figure out how to get that to display without being disruptive, and I'm glad I landed on a solution that feels natural while not requiring an explicit click (unless on a tablet). I'm also happy with how I was able to change the direction of the tree just by swapping out to a different view just by swapping out the stylesheet governing which the axes of the flexboxes in it.

One last thing to mention is how data is stored. Remixes are stored as objects nested inside a main object, and are accessed by using their project ID as a key, making it easy to locate information about a project (and its child remixes, which are stored in a list that gets dynamically filled as more remixes are added). Having a centralized object keep track of it all proved to be a robust and easily-used solution for accessing info on one project relative to others, which was essential for corralling all the asynchronous, isolated XHR jobs running at once.

I'm hoping to eventually work on this more to make it more useful and easy to use. Some features I'd like to add but didn't get to are:

  • Storing already-loaded remix info in localStorage
  • Next and previous page buttons when number of remixes shown is less than the number that actually exist
  • Options to sort projects by release date, views, loves, favorites, and remixes
  • A way to zoom the tree out and in so you can see the whole thing better
  • Ability to search the Scratch API by a keyword, then put that into the Remix Tree Visualizer (suggested by Nick during our Feedback Party)
